# 知识库如何节省 Token：实战演示

## 📊 场景设定

**问题**："项目中如何处理收益率曲线倒挂的数据？请详细说明整个流程，包括数据获取、处理逻辑和可视化实现。"

---

## 方式一：传统方式（无知识库）

### Token 消耗分析

假设项目中有一个 `yield-curve.js` 文件，包含约 2000 行代码：

#### ❌ 高 Token 消耗流程

```
用户提问
  ↓
【需要手动提供上下文】
  - yield-curve.js: 2000 行 ≈ 15,000 tokens
  - curve-analyzer.js: 800 行 ≈ 6,000 tokens
  - data-processor.js: 500 行 ≈ 4,000 tokens
  - README.md: 200 行 ≈ 1,500 tokens
  ↓
总计输入 tokens: ~26,500 tokens
  ↓
LLM 处理 + 生成回复: ~30,000 tokens
  ↓
总消耗: ~56,500 tokens
```

#### 💰 成本计算（以 GPT-4 为例）

- 输入: 26,500 tokens × $0.01/1K = **$0.265**
- 输出: 30,000 tokens × $0.03/1K = **$0.90**
- **总成本: $1.165**

#### ⚠️ 问题

1. **需要手动找到相关文件** - 浪费时间
2. **可能包含无关代码** - 浪费 token
3. **每次都要重新上传** - 重复消耗
4. **难以维护最新版本** - 过时上下文

## 方式二：知识库 RAG（推荐）

### Token 消耗分析

使用本地 Qdrant 知识库，已上传并索引项目文档：

#### ✅ 低 Token 消耗流程

```
用户提问
  ↓
【知识库检索】
  - 查询向量生成: ~50 tokens

  ↓
【仅返回相关内容】
  片段 1: "收益率曲线数据处理逻辑位于 yield-curve.js..."
  片段 2: "数据获取通过 FRED API..."
  片段 3: "可视化使用 Chart.js 渲染..."
  ↓
总计输入 tokens: ~1,500 tokens (3个相关片段 + 用户问题)
  ↓
LLM 处理 + 生成回复: ~2,000 tokens
  ↓
总消耗: ~3,500 tokens
```

#### 💰 成本计算

- 输入: 1,500 tokens × $0.01/1K = **$0.015**
- 输出: 2,000 tokens × $0.03/1K = **$0.06**
- **总成本: $0.075**

#### ✅ 优势

1. **自动找到相关内容** - 无需手动搜索
2. **只返回相关片段** - 精准高效
3. **智能去重** - 避免重复内容
4. **元数据过滤** - 可按类别/标签筛选

### 1. 检索增强生成 (RAG) 架构

```
┌─────────────────────────────────────────────────────┐
│                    用户问题                          │

                   │
                   ▼
┌─────────────────────────────────────────────────────┐
│              1. 向量化查询                            │
│      问题 → [0.23, -0.45, 0.67, ...]               │

└──────────────────┬──────────────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────────────┐

│              3. 组合上下文                            │

└─────────────────────────────────────────────────────┘
```

### 2. 为什么 RAG 更高效？

#### 传统方式的"冗余问题"

```javascript
// 假设整个文件 2000 行，你只需要其中 50 行

// ❌ 传统方式：必须发送整个文件（2000 行）
const YieldCurveProcessor = {
  // ... 1000 行无关代码 ...
  // ... 500 行辅助函数 ...
  // ... 450 行注释 ...

  // ✅ 你真正需要的部分（50 行）
  processInvertedYield(data) {
    // 处理收益率曲线倒挂
  },

  // ... 500 行更多无关代码 ...
}
```

**Token 消耗**: 2000 行 × 7.5 tokens/行 = **15,000 tokens**

#### RAG 方式的"精准定位"

```javascript
// ✅ 知识库方式：只返回相关片段

【检索结果 1】相关度: 0.95
"收益率曲线倒挂处理逻辑位于 yield-curve.js:

processInvertedYield(data) {
  // 1. 检测倒挂
  const isInverted = data.spread_10y_2y < 0;

  // 2. 触发警告
  if (isInverted) {
    this.triggerAlert();
  }

  // 3. 更新图表
  this.updateChart(data);
}
"
```

**Token 消耗**: **~200 tokens** (精准片段)

## 🎯 实战建议

### 1. 何时使用知识库？

✅ **推荐使用**：
- 项目文档超过 100 个文件
- 代码库 > 10,000 行
- 需要频繁查询历史文档
- 多人协作项目
- 需要跨文件关联分析

❌ **不需要使用**：
- 单个简单脚本 (< 100 行)
- 一次性查询
- 文档内容变化极快

### 2. 如何优化 Token 消耗？

#### 上传阶段

```bash
# ✅ 好的做法：按类别组织文档
knowledge-base/
├── project/       # 核心代码文档
├── research/      # 研究报告
└── business/      # 业务知识

# ❌ 避免：所有文件堆在一起
all-docs/
├── everything.md  # 100万行的巨型文件
```

#### 查询阶段

```python
# ✅ 使用元数据过滤（未来 Qdrant 支持）
results = kb.search(
    query="如何使用 API？",

        "tags": ["api", "frontend"] # 特定标签

)

# ❌ 避免模糊查询

    query="怎么用这个？"  # 太模糊，检索不精准
```
