# 面向宏观、行业与公司对比分析的金融看板：风格、架构与实现路径

## 摘要
本报告旨在为构建一个服务于宏观、行业与公司三层对比分析的非交互式、高信息密度金融看板，提供一套完整的设计方案与技术实现路径。报告融合了2024-2025年的最新技术趋势与实践研究，内容涵盖从设计哲学、数据架构到具体代码实施的全流程。核心目标是：在“技术从简、PC端优先、数据日内更新”的约束下，为个人零售交易员提供一个信息呈现清晰、可长时间专注、能辅助快速决策的静态分析工具。

---

## 1. 目标与约束条件定义
本看板的设计始于对核心目标与刚性边界的明确定义。

### 1.1 核心分析目标
1.  **宏观层面**：监控关键经济指标（如中美CPI、PMI、利率、M2）的趋势、拐点及其与市场宽基指数的相关性。
2.  **行业层面**：进行跨行业的截面比较与动态轮动分析，识别强势/弱势板块，评估行业对特定宏观因子的暴露度。
3.  **公司层面**：对比同行业或跨行业公司的核心财务指标、估值水平与市场表现，进行快速优劣筛查。

### 1.2 用户场景与核心约束
*   **用户**：经验丰富的个人零售交易员，具备独立分析能力。
*   **核心场景**：**仅查看与分析**，无需交互式筛选、下钻或配置。信息需预先计算、直接呈现。
*   **关键约束**：
    *   **技术从简**：避免复杂的前端框架（如React/Vue）和后端基础设施。优先采用原生技术、静态部署。
    *   **设备与性能**：为PC端大屏显示器优化，确保在高信息密度下渲染流畅。
    *   **数据时效**：实现**日内更新**，确保开盘前可获得前一日收盘后的完整数据（行情、财务、宏观）。

### 1.3 设计哲学
基于用户场景，确立 **“叙事优于探索，密度兼顾清晰”** 的核心设计哲学。看板不是一个等待查询的工具，而是一份**自动生成的每日分析报告**，所有关联、对比与结论应通过视觉编码直接传达。

## 2. 风格与设计原则
基于对高密度信息仪表板及2025年UI趋势的研究，本看板采用一种 **“理性实用主义”** 风格，在专业性与视觉舒适度间取得平衡，**明确摒弃过度装饰的新粗野主义，并审慎采用玻璃拟态元素**。

### 2.1 视觉层次：三层架构
采用 **“焦点层 - 分析层 - 上下文层”** 结构，引导用户视线，管理认知负荷。
*   **焦点层**：位于视觉中心（遵循F形模式）。使用**超大字体KPI**（如“沪深300”、“当日关键风险”）、**核心资金曲线图**或**市场情绪仪表**。目标：0.2秒内传递整体状态。
*   **分析层**：环绕焦点层。由**小型多图、矩阵热力图、平行坐标轴**等组成，用于解释宏观驱动、行业轮动、公司对比等具体分析议题。
*   **上下文层**：分布在边缘。以**高密度表格、小字号标签、完整图例、数据更新时间戳**等形式提供所有支持性细节和元数据，供深度核查。

### 2.2 布局模式：全景矩阵与焦点扩散
针对PC端单屏，推荐两种核心布局，可结合使用：
1.  **全景矩阵式布局**：将屏幕划分为均匀网格（如4x4），每个单元格承载一个分析视图（如一个行业的迷你雷达图或K线）。通过单元格底色（如浅绿/浅红）进行快速模式识别，适用于行业监控。
2.  **焦点扩散式布局**：中心放置核心结论图（如股债性价比），周围环绕多个关联的“卫星”图表（如利率趋势、信用利差、资金流向），通过共享的图例或坐标轴直观展示逻辑关联，适用于宏观主题分析。

### 2.3 色彩与视觉编码系统
*   **基础色调**：采用**深色主题**作为基准，主背景色为深灰（`#121212`或`#1E1E1E`），以减轻长时间观看的视觉疲劳，并使数据色彩更突出。
*   **功能色（语义色）**：建立严格的、无障碍的功能色彩系统。
    *   **双极性数据（涨跌）**：采用**发散色板**。例如，下跌使用`#EF5350`（红）到`#B71C1C`（深红）的渐变；上涨使用`#66BB6A`（绿）到`#1B5E20`（深绿）的渐变；中性点（如零轴）使用`#78909C`（中灰）。
    *   **风险等级**：采用**明度阶梯**，如关注（`#FFF9C4`）、警告（`#FFB74D`）、严重（`#E53935`），避免持续使用高饱和警报色。
    *   **分类数据**：使用**定性色板**（如Tableau Classic 10），确保类别间可区分。
*   **CSS变量化设计令牌**：所有颜色、间距、圆角、阴影均通过CSS自定义属性定义，实现主题一键切换与全局一致性。
    ```css
    :root {
      --color-surface-primary: #1e1e1e;
      --color-text-primary: rgba(255, 255, 255, 0.87);
      --color-data-negative: #EF5350;
      --color-data-positive: #66BB6A;
      --spacing-unit: 8px;
      --border-radius-sm: 4px;
    }
    ```

### 2.4 排版与密度控制
*   **字体**：界面文字使用无衬线字体（如**Inter, Roboto**），**数字、代码、股票代码强制使用等宽字体**（如**Roboto Mono, SF Mono**），确保数据列严格对齐。
*   **栅格系统**：所有组件尺寸、间距均基于**4px或8px的基础单元网格**，实现严谨的视觉对齐。
*   **对抗视觉疲劳**：
    *   使用**精细的分隔线**（`1px solid rgba(255,255,255,0.1)`）和**一致的背景色块**来划分区域，而非大量留白。
    *   动态数据更新采用**定向微动画**（如数值从底部淡入），避免使用闪烁。
    *   严格统一字重（Regular, Medium, Semibold），创建清晰的文本层级。

## 3. 核心分析模块与视图规划
将三大分析目标分解为可执行的仪表板视图，并规划具体的图表组合。

### 3.1 宏观经济仪表板
*   **用途**：展示经济增长、通胀、货币、情绪等维度的关键指标趋势及相互关系。
*   **视图规划**：
    1.  **焦点层**：**“宏观经济状态仪”**，结合多个指标合成的扩散指数，以仪表盘形式显示经济周期位置（衰退/复苏/过热/滞胀）。
    2.  **分析层**：
        *   **小型多图序列**：展示CPI、PPI、PMI、社会融资规模等关键指标的月度趋势线。
        *   **相关性热力图**：展示各宏观指标与主要股指（沪深300、标普500）在不同时间窗口下的滚动相关性。
        *   **平行坐标轴图**：连接“利率”、“通胀”、“汇率”等宏观因子与“消费”、“科技”、“金融”等行业的预期表现，展示因子暴露。
    3.  **上下文层**：最新数据发布日历、指标定义说明、数据来源及更新时间。

### 3.2 行业对比矩阵
*   **用途**：横向比较各行业表现，识别强势板块与轮动信号。
*   **视图规划**：
    1.  **焦点层**：**“行业轮动热力图”**。Y轴为行业列表（按当日涨跌幅排序），X轴为时间（最近20个交易日），颜色强度代表每日相对强弱。一眼可辨趋势持续性。
    2.  **分析层**：
        *   **聚类矩阵热力图**：行业x指标矩阵，指标可包括“近一月涨幅”、“PE分位数”、“营收增速”、“波动率”等。**对行和列进行聚类分析**，自动将相似的行业和相关的指标聚集，揭示板块族群特征。
        *   **改进型雷达图群**：采用**小型多图**形式，为每个行业绘制一个独立的标准化雷达图，展示其在“估值”、“成长”、“动量”、“质量”、“规模”五个维度的相对位置。按综合得分排列。
        *   **动态气泡图**：展示行业轮动。X轴和Y轴代表两个核心驱动因子（如“预期增长率”与“风险溢价”），气泡大小代表行业市值，颜色代表板块。配合时间轴动画，展示轮动轨迹。
    3.  **上下文层**：行业分类标准、成分股列表、各指标的计算公式与数据截止日。

### 3.3 公司财务快照与对比
*   **用途**：快速扫描与对比目标公司的核心基本面与市场表现。
*   **视图规划**：
    1.  **焦点层**：**“公司对比摘要栏”**。以紧凑表格形式，并列显示3-5家公司的核心KPI：市值、股价、PE(TTM)、PB、股息率、近一年涨跌幅。
    2.  **分析层**：
        *   **财务指标对比条形图**：对比ROE、毛利率、净利率、负债率等关键盈利能力与财务健康度指标。
        *   **估值趋势图**：叠加显示各家公司的历史PE Band，判断当前估值所处历史百分位。
        *   **价格相关性矩阵**：展示所选公司股价之间的近期相关性热力图，分析联动性。
    3.  **上下文层**：公司详细代码、所属行业、财报发布日期、数据来源标注。

## 4. 数据集成与更新机制
这是实现“日内更新”与“技术从简”目标的核心环节。方案基于 **“轻量级代理 + 自动化工作流 + 静态文件输出”** 的架构。

### 4.1 数据源选型建议
基于覆盖、稳定、免费的优先级：
1.  **A股核心数据源**：**Tushare Pro**。推荐理由：数据标准化程度高，覆盖行情、财务、宏观、新闻，文档完善，社区活跃。免费版足以启动，通过积分策略可满足日内低频更新需求。
2.  **补充与备用源**：**AkShare**。作为Tushare的补充或免费替代，覆盖范围极广，无积分限制，但需注意API稳定性和数据格式一致性。
3.  **全球宏观专项源**：**FRED API**。获取美国及全球权威宏观经济数据的标准，免费稳定，但更新有延迟。
4.  **前端图表库**：**Lightweight Charts by TradingView**（极致性能）或**Apache ECharts**（按需引入，功能全面）。两者均能完美满足K线、面积图、热力图等需求。

### 4.2 数据更新自动化工作流（核心机制）
采用 **GitHub Actions** 作为免费的自动化调度中心，替代传统服务器Cron Job。
*   **流程**：
    1.  **定时触发**：GitHub Actions 配置 `schedule`，每日收盘后（如 UTC 18:00）触发工作流。
    2.  **执行环境**：工作流在一个干净的 `ubuntu-latest` 运行器中启动。
    3.  **数据抓取与处理**：运行器执行一个 **Node.js/Python 脚本**。该脚本：
        *   通过**轻量级代理层**（封装在脚本内）调用 Tushare/AkShare API，规避前端CORS限制。
        *   获取原始数据，进行清洗、转换、标准化（如统一代码、单位换算、计算衍生指标）。
        *   将处理后的数据按看板视图所需的结构，输出为静态 **JSON 文件**（如 `data/macro.json`, `data/industries.json`）。
        *   同时生成一个 `meta.json`，包含本次更新的时间戳、数据哈希和状态。
    4.  **提交与部署**：脚本使用 **GitHub CLI (`gh`) 或个人访问令牌**，将生成的JSON文件提交并推送回本仓库。
    5.  **触发页面更新**：推送操作自动触发静态站点（GitHub Pages）的重新部署，更新后的数据随之上线。

### 4.3 前端数据状态管理
*   **数据加载**：前端HTML页面通过 `fetch()` 直接加载位于同一仓库下的 `data/*.json` 和 `meta.json` 文件。
*   **状态指示器**：解析 `meta.json` 中的时间戳，在页面角落显示“数据更新于：YYYY-MM-DD HH:MM”。
*   **缓存与回滚**：Git历史自然形成数据版本库。可通过Git标签标记重要数据版本，必要时回滚。

### 4.4 关键技术挑战与解决方案
*   **CORS问题**：所有数据获取逻辑必须封装在自动化工作流脚本或一个独立的无服务器函数中，前端绝不直接调用外部API。
*   **代码映射**：建立并维护一个 `symbol_mapping.json` 文件，统一不同数据源的股票代码、指标ID。
*   **API速率限制**：在抓取脚本中设计请求队列和延时，并利用本地缓存避免重复请求相同数据。

## 5. 简单但现代的技术栈建议
遵循“技术从简”原则，推荐一套零框架、高性能、易维护的前端技术组合。

### 5.1 核心架构：原生ES6模块化项目
采用基于 **Vite** 构建的纯原生 JavaScript (ES6+) 项目结构。Vite 提供极速的开发服务器和优化的构建输出，无需配置复杂的Webpack。

*   **项目结构示例**：
    ```
    src/
    ├── features/          # 按特性/视图组织，高内聚
    │   ├── macro/
    │   │   ├── view.js       # 宏观视图控制器
    │   │   ├── charts.js     # 宏观图表渲染器
    │   │   └── data-source.js # 宏观数据适配器
    │   ├── industry/
    │   └── company/
    ├── core/              # 应用核心基础设施
    │   ├── event-bus.js   # 基于EventTarget的简单事件总线
    │   ├── stores/        # 存储模式状态管理
    │   │   └── selectedStockStore.js
    │   └── utils.js
    ├── shared/            # 共享组件
    │   └── components/
    │       └── base-chart.js # 封装的通用图表组件
    ├── services/          # 外部服务交互层（前端用于加载静态JSON）
    ├── assets/
    │   └── styles/
    │       ├── tokens.css    # CSS设计令牌
    │       ├── base.css      # 基础样式
    │       └── views/        # 视图特定样式
    └── main.js            # 应用入口，动态导入视图
    ```

### 5.2 状态与通信：存储模式与自定义事件
*   **共享状态**：采用**存储模式**。例如，`selectedStockStore.js` 导出一个可观察状态对象和修改函数，内部使用 `EventTarget` 通知变更。
    ```javascript
    // stores/selectedStockStore.js
    let currentStock = null;
    const eventBus = new EventTarget();
    export const stockStore = {
      get() { return currentStock; },
      set(newStock) {
        currentStock = newStock;
        eventBus.dispatchEvent(new CustomEvent('stockchanged', { detail: newStock }));
      }
    };
    ```
*   **模块通信**：各视图模块通过**订阅自定义事件**实现解耦联动。

### 5.3 图表库集成
*   **选择**：**Lightweight Charts** 用于核心K线/行情图；**Apache ECharts**（按需引入）用于热力图、雷达图、平行坐标轴等复杂图表。
*   **集成方式**：在 `shared/components/base-chart.js` 中封装图表初始化、更新、销毁的生命周期管理，确保内存不泄漏。
*   **性能**：利用 `Intersection Observer API` 实现图表的懒加载，仅当图表进入视口时才进行初始化。

### 5.4 样式管理
*   **CSS变量**：所有设计决策定义在 `tokens.css` 中。
*   **CSS Modules**：通过Vite原生支持，为每个组件或视图提供作用域样式，防止冲突。

## 6. 实施路线图与关键任务
一个从零开始、分为六个阶段的可执行路线图。

### 阶段一：项目初始化与环境搭建 (预计: 2天)
1.  **创建代码仓库**：在GitHub创建新仓库，启用GitHub Pages。
2.  **搭建本地开发环境**：
    *   `npm create vite@latest financial-dashboard -- --template vanilla`
    *   安装必要依赖：`echarts`， `lightweight-charts`。
    *   配置 `vite.config.js` 以优化构建。
3.  **设计令牌与基础样式**：编写 `tokens.css` 和 `base.css`，确立深色主题与设计系统。
4.  **搭建基础HTML骨架**：创建 `index.html`，规划主要布局区域（焦点层、分析层、上下文层）。

### 阶段二：数据管道与自动化工作流搭建 (预计: 5天)
1.  **申请与测试数据源API Key**：注册Tushare Pro、FRED等，验证数据获取。
2.  **开发数据获取与处理脚本**：
    *   创建 `scripts/fetch-data.js`（Node.js）。
    *   实现从Tushare等获取数据、清洗、转换的逻辑。
    *   定义并输出标准化的 `data/*.json` 和 `meta.json`。
3.  **配置GitHub Actions工作流**：
    *   创建 `.github/workflows/update-data.yml`。
    *   配置定时任务、安装依赖、运行脚本、自动提交的完整流程。
    *   测试工作流能否成功运行并更新仓库文件。
4.  **实现前端数据加载模块**：创建 `services/data-loader.js`， 使用 `fetch` 加载本地JSON。

### 阶段三：核心可视化组件与视图开发 (预计: 10天)
1.  **封装基础图表组件**：开发 `shared/components/base-chart.js`， 统一管理ECharts/Lightweight Charts实例。
2.  **实现宏观仪表板视图**：
    *   开发 `features/macro/view.js` 及相关模块。
    *   集成“宏观经济状态仪”、小型多图序列、相关性热力图。
3.  **实现行业对比视图**：
    *   开发 `features/industry/view.js`。
    *   集成聚类矩阵热力图、行业轮动热力图、小型雷达图群。
    *   实现行业聚类算法或集成简单库。
4.  **实现公司对比视图**：
    *   开发 `features/company/view.js`。
    *   集成对比摘要栏、财务指标条形图、估值趋势图。
5.  **开发共享状态存储**：实现 `core/stores/` 下的相关存储，用于连接不同视图。

### 阶段四：集成、布局与样式优化 (预计: 5天)
1.  **视图集成与路由**：在 `main.js` 中实现简单的标签页或视图切换逻辑，动态加载不同特性模块。
2.  **应用整体布局**：使用CSS Grid/Flexbox，将各视图模块嵌入到第2章规划的布局模式中。
3.  **精细化样式调整**：确保所有图表、表格、文字在深色主题下清晰可读，符合无障碍标准。
